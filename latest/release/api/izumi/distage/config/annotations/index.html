  2.  final class AutoConf extends Annotation with ConfiguratorAnnotation

    Inject parameter from configuration.

    Inject parameter from configuration.

    The config path is derived automatically from the context.

    For example, given a class:

    class Service(@AutoConf hostPort: HostPort)

    where:

    case class HostPort(host: String, port: Int)

    you'll be able to define configuration like this:

    Service {
  HostPort {
    host = "example.com"
    port = 8080
  }
}

    If the requesting class is bound as named binding, you can disambiguate the config path to uniquely configure that class:

    new ModuleDef {
  make[Service].named("namedService")
}

    Config:

    Service {
  namedService {
    HostPort {
      host = "example.com"
      port = 8080
    }
  }
}

    If a class requests several values of the same type from config, you can disambiguate the config path using parameter names:

    class Service(@AutoConf param1: HostPort, @AutoConf param2: HostPort)

    Config:

    Service {
  HostPort {
    param1 {
      host = "example.com"
      port = 8080
    }

    param2 {
      host = "google.com"
      port = 80
    }
  }
}

    Specifically, the path is defined by segments:

    {binding_type}.{binding_id_key?}.{config_type}.{parameter_name?}

    Where segments marked with ? at the end are optional.

    Use this annotation when you need to inject one config section into one component

  5.  final class Conf extends Annotation with ConfiguratorAnnotation

    Inject parameter from config value at the specified path.

    Inject parameter from config value at the specified path.

    For example, given a class:

    class Service(@Conf("cassandra") cassandraHost: HostPort)

    where:

    case class HostPort(host: String, port: Int)

    you'll be able to define configuration like this:

    cassandra {
  HostPort {
    host = "cassandra"
    port = 9003
  }
}

    Note that you may also omit the name of the case class:

    cassandra {
  host = "cassandra"
  port = 9003
}

    Specifically, this annotation differs from AutoConf only in that the binding type and binding id components are replaced by explicitly defined path, so that the whole path is defined by segments:

    {path}.{config_type?}.{parameter_name?}

    Where segments marked with ? at the end are optional.

    Use this annotation when you want to share one config section between several components

  7.  final class ConfPath extends Annotation with ConfiguratorAnnotation

    Inject parameter from config value at the specified fully qualified path.

    Inject parameter from config value at the specified fully qualified path.

    For example, given a class:

    class Service(@ConfPath("service.conf" hostPort: HostPort)

    where:

    case class HostPort(host: String, port: Int)

    you'll be able to define configuration like this:

    service {
  conf {
    host = "example.com"
    port = 8080
  }
}

    Note, you will NOT be able to specify the type of the case class:

    service {
  conf {
    HostPort /* ILLEGAL */ {
      host = "example.com"
      port = 8080
    }
  }
}

    Use this annotation if you are too lazy to learn how AutoConf or Conf work
